# Data Cleaning & Validation Pipeline

## Context

Dataset is generated by a weak local model. Each row in the JSONL represents one source function from a SCADA library, decomposed into testable tasks. Quality is poor — test cases are trivial or wrong, process descriptions are inconsistent, format violations are common.

Goal: clean and validate every task, output one task per row in a new JSONL.

---

## Original JSON Schema (per row)

```json
{
  "source_function": "str",        // the SCADA library function being targeted
  "refused": "bool",               // true = generation failed, skip entire row
  "original_description": "str",   // large text from which tasks were parsed
  "tasks": [
    {
      "task": "str",               // general natural language description of the task
      "processes": [
        {
          "description": "str",    // description of a single .c file to be written
          "execution_order": "int" // 1 = run first, 2 = run second
        }
      ],
      "target_function": "str",    // same as source_function, may be absent in newer data
      "test_cases": [
        {
          "input": ["str"],        // flat list: [type1, val1, type2, val2]
          "expected_output": "str" // 2 newline-separated tokens
        }
      ]
    }
  ]
}
```

### Test Case Format Detail

- Input is always length 4: `[type1, val1, type2, val2]`
- `type` is `i` (32-bit integer) or `s` (4-char string, max 4 bytes)
- `val1` is written to file `2001`, `val2` is written to file `2002`
- After processes run, harness reads `2001` and `2002` back
- `expected_output` = two tokens joined by `\n`, in same type order as input
- I != O is required — if input equals output, a silent runtime failure will pass undetected

### Processes Detail

- Each task has 1 or 2 processes (C files, each with their own `main`)
- `execution_order=1` runs first, `execution_order=2` runs second
- Two processes needed when task involves an event listener — process 1 sets up the listener, process 2 sends data to it
- Files `2001` and `2002` are not real filesystem files — they are daemon-emulated sensor records (SCADA IO), record 0 is 4 bytes

---

## Input / Output

- **Input:** raw JSONL, one row per source function
- **Output:** `_cleaned.jsonl` — one row per task (flattened from source rows)
- **Side output:** `_quarantine.jsonl` — tasks that failed all fix attempts, with logged failure reason

---

## Phase 1 — Structural Cleaning (per row)

**Row-level:**
1. Skip entire row if `refused: true`

**Task-level (per task in row):**
2. Drop task if process count > 2
3. Drop task if `execution_order` values are not exactly `{1}` or `{1,2}`
4. Drop task if `execution_order` has duplicates
5. Drop entire row if no tasks remain after drops

**Test case-level (per test case in task):**
6. Drop duplicate test cases within a task
7. Flag for LLM fix:
   - Input length != 4
   - Type values not `i` or `s` (case-sensitive)
   - String value longer than 4 characters
   - Integer value outside signed 32-bit range (-2147483648 to 2147483647)
   - `expected_output` does not split into exactly 2 tokens on `\n`
   - Output token type inconsistent with input type (e.g. input type `i` but output token is not parseable as integer)
   - Input values == output values (I == O, both tokens identical to input values)

---

## Phase 2 — LLM Test Case Fix (per flagged test case)

**Prompt contains:**
- `source_function`
- `task` description
- `processes` descriptions
- The broken test case
- Exact reason it was flagged

**Ask LLM to:** generate a replacement test case where:
- Input and output are structurally valid
- I != O (transformation must be non-trivial)
- Output types match input types
- Values are within range

**After fix:**
- Re-run structural validation on new test case
- Also update process descriptions to reflect the new test case if values changed significantly
- Retry up to N times
- If still broken after N retries → quarantine the task, log reason

---

## Phase 3 — Code Generation & Validation (per task)

### Setup

Each task has 1 or 2 processes. Each process description describes one `.c` file with its own `main`. The master agent generates C code for each process independently.

### Tools available to master agent:
- **Compile C code tool** — checks if generated code compiles, returns errors
- **Specialist agent tool** — master calls specialist with a question; specialist has RAG access to SCADA library docs/source and returns a focused answer (keeps master agent context clean)

### Code generation loop (per process):
1. Master agent generates C code from process description
2. Run compile check
3. If compile fails → master agent reads errors, optionally queries specialist, fixes code, retry
4. Repeat up to N compile attempts
5. If compile never succeeds → quarantine task

### Runtime validation (after all processes compile):
Run harness with all test cases for the task:
- Harness writes input values to files `2001` and `2002`
- Runs process(es) in execution order
- Reads files `2001` and `2002` back
- Compares read values to `expected_output`

### Runtime failure resolution (in order):

**Stage A — Fix process description (up to N attempts):**
- Show LLM: current process description + test case IO
- Ask: rewrite process description so that code implementing it would produce the expected output
- Regenerate C code from new description → recompile → rerun harness
- Goal: keep test case as ground truth, fix description to match

**Stage B — Fix IO (if Stage A exhausted):**
- Show LLM: process description + what code actually produces vs what was expected
- Ask: rewrite test case IO so it matches what the process logically does
- Regenerate C code → recompile → rerun harness
- Goal: keep process description as ground truth, fix test case to match

**Stage C — Quarantine (if Stage B exhausted):**
- Write task to `_quarantine.jsonl`
- Log: all attempted descriptions, all attempted test cases, all compile errors, all runtime outputs

## Output Row Schema (`_cleaned.jsonl`)

One task per row:

```json
{
  "source_function": "str",
  "original_description": "str",
  "task": "str",
  "processes": [
    {
      "description": "str",
      "execution_order": 1
    }
  ],
  "test_cases": [
    {
      "input": ["i", "32", "s", "abcd"],
      "expected_output": "33\nbcde"
    }
  ]
}
```

  "phase": "1 | 2 | 3",
  "attempts": [...]
